# Development guide

This is the practical "how to work on it" page: set up the environment,
regenerate code from the contract, build, test, and follow the branch/PR
workflow. For *why* the project is shaped this way, read the
[Architecture Decision Records](./adr/README.md); for *what* you are building,
read the [architecture overview](./architecture/overview.md).

## Prerequisites

You need **Docker** and an editor or CLI that understands the
[Dev Containers spec](https://containers.dev/) (for example, VS Code with the Dev
Containers extension, or the `devcontainer` CLI). You do **not** need Go, Rust,
or `protoc` on your host - that is the entire point of the container
([ADR-0003](./adr/0003-dev-container-environment.md)).

## Set up the environment (Dev Container)

The repository carries `.devcontainer/Dockerfile` and
`.devcontainer/devcontainer.json`. Opening the folder in a Dev Container builds
an image with the full, pinned toolchain:

- Go (from the `devcontainers/go:1-bookworm` base image)
- Rust **1.82.0** (via rustup, minimal profile) with `clippy` and `rustfmt`
- `protoc` (`protobuf-compiler`)
- `protoc-gen-go`, `protoc-gen-go-grpc`, and `buf`

On creation, the container runs a `postCreateCommand` that prints the versions of
`go`, `cargo`, `protoc`, `protoc-gen-go`, and `buf`. If any of those are missing,
the container build is the thing that fails - loudly, and before you waste time
debugging a "works on my machine" mystery.

Ports `2222` (the SSH gateway) and `50051` (the gRPC engine) are forwarded to the
host so you can reach the running system during development.

> Why a container instead of host installs? Reproducibility and CI parity. One
> pinned toolchain, identical for every contributor and (intended to be)
> identical to CI, deletes a whole class of version-mismatch bugs. Full reasoning
> in [ADR-0003](./adr/0003-dev-container-environment.md).

## Regenerate code from the contract

`proto/poker/v1/poker.proto` is the single source of truth for the Go<->Rust
boundary. Both sides generate code from it; you **start any contract change by
editing the proto**, then regenerate (see
[spec-driven development](./practices/spec-driven-development.md)).

```bash
# Regenerate everything (Go + Rust):
make proto

# Just the Go stubs (protoc-gen-go / protoc-gen-go-grpc into gateway/internal/pokerpb):
make proto-go

# The Rust side is generated by engine-service/build.rs during the cargo build:
cargo build -p engine-service     # equivalently: make proto-rust
```

The Rust code is produced by `tonic-build` at compile time (so building the crate
*is* regenerating it). The Go code is produced by `protoc` and is **committed** to
the repo under `gateway/internal/pokerpb/` - which is why CI runs a
**codegen-drift check** (regenerate, then fail on `git diff`). If you change the
proto, regenerate and commit the generated Go alongside it, or CI will catch the
drift ([ADR-0007](./adr/0007-ci-cd-pipeline.md)).

## Build

```bash
# Build both halves (Rust workspace + Go gateway):
make build

# Or individually:
make build-rust        # cargo build --workspace
make build-go          # runs proto-go, then: cd gateway && go build ./...
```

## Test

```bash
# Run both test suites:
make test

# Or individually:
make test-rust         # cargo test --workspace
make test-go           # runs proto-go, then: cd gateway && go test ./...
```

The testing approach is a layered pyramid across both languages (Rust unit/engine
-> Go unit -> gRPC integration -> SSH end-to-end). See
[ADR-0006](./adr/0006-testing-strategy.md) and
[test-driven development](./practices/test-driven-development.md) for what belongs
at each layer.

## Quality checks (what CI gates on)

```bash
make lint-rust         # cargo clippy --workspace --all-targets -- -D warnings
make fmt-rust          # cargo fmt
cd gateway && go vet ./...
```

`clippy -D warnings`, `cargo test`, and the Go build/vet/test are merge gates,
each under a job timeout so a hang fails fast rather than wedging the runner.
Formatting is advisory for now. Full pipeline detail:
[ADR-0007](./adr/0007-ci-cd-pipeline.md).

## The contribution workflow

The whole project runs on one simple rule, mirrored on
[GitHub Project board #2](./roadmap.md):

> **One issue = one branch = one PR.**

Concretely:

1. **Pick one issue** from the board (M1-M12). Keep the scope to that single
   issue - small, reviewable changes.
2. **Create a branch** named `m<phase>/<slug>`, where `<phase>` is the milestone
   number and `<slug>` is a short kebab-case description. Examples:
   - `m2/proto-schema`
   - `m3/apply-action-rpc`
   - `m6/table-fan-out`
3. **Make the change**, including regenerating code if you touched the proto, and
   add tests at the appropriate layer of the pyramid.
4. **Open a PR targeting `main`.** The PR description says **`Closes #N`** (so
   merging closes the issue), and the PR stays attached to project #2.
5. **CI must pass** - Rust `clippy -D warnings` + tests, Go build/vet/test, and
   the codegen-drift check - before merge.

This keeps every commit internally consistent (the monorepo lets a contract
change and both generated halves land together -
[ADR-0005](./adr/0005-monorepo-structure.md)) and keeps history readable: each PR
maps to exactly one tracked decision or feature.

## A typical change, end to end

Adding a new field to the contract is the canonical example because it touches
every part of the workflow:

1. Branch: `git switch -c m2/seatview-timebank`.
2. Edit `proto/poker/v1/poker.proto` (add the field).
3. Regenerate: `make proto` (updates Go stubs and rebuilds Rust codegen).
4. Implement the Rust side in `engine-service`, the Go side in `gateway`, with
   tests.
5. Run `make build && make test && make lint-rust`.
6. Commit the proto change **and** the regenerated Go code together.
7. Open the PR: title describing the change, body with `Closes #N`, attached to
   project #2; let CI run.
